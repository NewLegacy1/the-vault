# Stationarity & Era-Splitting for Small Intraday Event Studies (40–200 Events)

Motivated by the DeltaTrend ES-sweep lesson (a setup significant in 2023 but dead when pooled 2022–25 — pooling across eras can both manufacture and hide edges) and by the Vault's own Track-B protocol, which reports full-sample and OOS EV but has no formal era test. Question: which stationarity tests and splitting protocols actually work at n = 40–200 events?

## Key findings

- **CLAIM (test taxonomy, MetricGate/strucchange lineage):** the classical toolkit divides by what you know: (1) **Chow test** — F-test comparing pooled vs split regressions at a *pre-specified* break date; high power if the date is right, near-zero if wrong. (2) **sup-F / Quandt** — Chow computed at every interior candidate date, take the max; estimates an unknown break date but pays a multiplicity price (critical values are larger). (3) **CUSUM** (recursive residuals) — a "metal detector" for drift with no date hypothesis, better for gradual erosion than sharp breaks. Rule of thumb from the source: "Chow is a scalpel; CUSUM is a metal detector."
- **CLAIM (OOS-failure diagnosis, aligrithm):** an OOS failure has four distinct causes — (1) no edge ever (overfit), (2) leakage, (3) sample too small, (4) **regime change between IS and OOS** — and the fourth is routinely mislabeled "overfit." The proposed cheap sequential diagnosis: permutation-test the IS result → audit the pipeline for look-ahead → check sample size → then run a **regime-overlap test**: characterize IS and OOS windows by structural features (rolling realized vol, autocorrelation, cross-asset correlation, trend strength) and test distributional distance (per-feature KS tests; joint energy distance). Low overlap = the OOS lived in a different regime; the verdict is "untested in this regime," not "falsified."
- **CLAIM (era-pooling arithmetic, general):** if an event has EV \(+x\) in era A (n₁ events) and ~0 in era B (n₂), the pooled estimate is \(x \cdot n_1/(n_1+n_2)\) with inflated variance — pooling *dilutes* a real era-bound edge toward insignificance (the ES-sweep case), and conversely a lucky era inside a pooled window can *manufacture* a "significant" full-sample edge. Either direction, the pooled number answers the wrong question. The right object at Stage-0 is the **era-conditional EV vector**, not the pooled scalar.
- **Power reality at n = 40–200 (the operative constraint):** a Chow-type mean-shift test on trade EVs with per-trade σ ≈ 1R detects only shifts of ≳ 0.5–0.8R per trade at n=100 split 50/50 (two-sample t power arithmetic). Subtle era decay is **undetectable** at Vault sample sizes; formal tests can only catch gross breaks. Consequence: the protocol must lean on *design* (pre-specified era blocks, rolling windows, regime-overlap checks) rather than on significance tests.
- **CLAIM (monitoring after promotion, kniyer/quantbeckman):** for live/walk monitoring, CUSUM with the reference mean μ₀ taken from a *clean validation period* (never the full backtest — that includes the regime you're trying to detect) accumulates evidence of decay; Page-Hinkley variant adapts its reference for slow erosion; rolling Sharpe with confidence bands is the intuitive fallback. Bayesian online changepoint detection is the heavy tool; its useful prior heuristic: assume ~3 regimes per dataset length, so a 3-year daily sample implies expected regime length ≈ 1 year — consistent with treating **calendar years as the default era block**.

## Details / mechanics

**Era-splitting protocol for a 40–200-event Stage-0 study (concrete, pre-registered):**

1. **Default blocks: calendar years** (or halves if the sample spans < 3 years). Never let the data choose the split (that's sup-F, which needs its own multiplicity correction and more n than we have).
2. **Report the era vector:** per block — n, EV, bootstrap 95% CI, WR, avg R. A candidate "passes era consistency" if **no block's CI sits wholly below zero** and the sign of EV agrees in blocks holding ≥ 25% of events. (Weaker than demanding significance per block — impossible at these n — but strong enough to catch the ES-sweep pattern of one hot year carrying a dead pool.)
3. **Rolling-window EV curve:** EV over a sliding window of ~30 events stepped by ~10, plotted against event date. Purely descriptive — no test — but it is the single most informative picture for "2023-only" edges, and it costs one loop in the analysis script.
4. **Regime-overlap check between IS and OOS:** before reading an OOS number as confirmation/falsification, KS-test 2–3 structural features (20-day realized vol; VIX band occupancy; trend-day fraction) between the IS and OOS windows. If distributions differ materially, label the OOS verdict "regime-shifted" in the scorecard rather than counting it as a clean kill or confirm.
5. **Gross-break test (only when the era vector already looks broken):** two-sample t / Mann-Whitney between the suspect block and the rest, Bonferroni-corrected by the number of blocks. This is confirmatory decoration; the decision weight stays on the era vector and the CI-based kill rules already in `kill-lessons-track-b.md` (#13).
6. **Pooling rule:** pooled EV is quoted *only when* the era vector passes step 2. If it fails, the honest statement is "edge observed only in era X; treat as untested elsewhere" — which for prop purposes usually means kill or shelve, since next month is always a new era draw.

**For the Dual46 walk (monitoring, not testing):** May and June are natural era blocks. The Beta-binomial note (topic 44) pools them for the WR posterior *conditional on* this note's check: if June-vs-May WR or EV differs grossly (e.g. non-overlapping bootstrap CIs), discount the earlier month's pseudo-counts (power prior δ ≈ 0.5) instead of pooling at full weight. A simple CUSUM on trade R-multiples, referenced to the June replay mean, is the cheapest live decay alarm and can run in a spreadsheet column: \(S_t = \max(0, S_{t-1} + (\mu_0 - r_t) - k)\) with slack k ≈ 0.25R; alert if \(S_t\) exceeds ~4–5.

## APPLICATION TO THE VAULT

1. **Extend `analyze-event-study.ts` output (post-May backlog, small):** add (a) per-calendar-year era vector (n, EV, CI), (b) the 30-event rolling-EV series, (c) an era-consistency verdict line. Steps 1–3 above are ~40 lines of TypeScript on data the script already holds; they turn the ES-sweep failure mode into a default-visible diagnostic at Stage-0.
2. **Scorecard field:** `eraConsistency: PASS | FAIL(block) | THIN` beside full-sample EV. Kill-lessons rule #13 (negative full EV, CI excl. 0 → kill) stays supreme; era consistency adds the *symmetric* guard — positive pooled EV carried by one block → BLOCK, not promote.
3. **Reclassify, don't retune:** when OOS fails but the regime-overlap check shows the OOS window was structurally different, the correct ledger entry is "untested in regime Y" — which still means no promotion (hard stops unchanged), but it stops the kill-lesson file from encoding "falsified" claims the data doesn't support.
4. **Walk monitoring line (immediate, zero code):** add the CUSUM column above to the May-walk ledger with μ₀ from June replay. It will be noisy at n=15 — its job is not significance, it is forcing the decay conversation onto a number instead of a feeling.
5. **Accept the power limit explicitly:** at Vault sample sizes, formal break tests detect only catastrophic decay. The defense is structural: pre-registered eras, rolling pictures, regime tags (topic 45), and the multiple-testing budget (topic 51) — not p-values.
