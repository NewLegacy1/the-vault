# Regime-Switching Monte Carlo: Markov Transition Matrices vs i.i.d. Reshuffle

Implementation-level follow-up to `deltatrend-monte-carlo-markov-prop-convexity.md` (which flagged regime-switching MC as the one genuinely new technique) and companion to `monte-carlo-prop-firm-survival.md`. Question: when does i.i.d. trade reshuffling mislead, what exactly does a Markov-regime MC do differently, and what is the minimal upgrade path for the Vault's TypeScript engine (`vault-app/lib/monte-carlo.ts`)?

## Key findings

- **CLAIM (canonical):** the Markov regime-switching model is Hamilton (1989, *Econometrica*): an unobserved discrete state \(s_t \in \{1..k\}\) follows a first-order Markov chain with row-stochastic transition matrix \(P_{ij} = P(s_t = j \mid s_{t-1} = i)\); each state carries its own outcome distribution (mean/variance). Estimation is by maximum likelihood via the Hamilton filter (or EM). For a 2-state model the matrix has only **2 free parameters** (\(p_{11}, p_{22}\); off-diagonals follow) — this is why it is feasible even on small ledgers.
- **EVIDENCE (controlled coverage study, 2026):** a 6,000-experiment study of bootstrap confidence intervals under known truth (marketmaker.cc, "Do Bootstrap Confidence Intervals for Backtest Statistics Cover?") found that for **maximum drawdown**, i.i.d. resampling is optimistic on *every* process tested — and critically, **under regime-switching volatility, block bootstrap fixes essentially nothing**: the nominal-5% worst-case drawdown had realized exceedance 0.165→0.168 (T=250) and 0.134→0.133 (T=1000) going from i.i.d. to block methods. Reason: "the blocks are far shorter than the regimes, so no resampling scheme … reproduces the possibility that the next path spends most of its life in the high-volatility state." Sharpe-type (mean) statistics were barely hurt by vol regimes; **path/drawdown statistics are exactly where regimes bite** — and path statistics are what the Vault's `E[$/wk]`-first hierarchy runs on.
- **EVIDENCE (same study):** even on truly i.i.d. Gaussian returns the bootstrap 5% worst-case drawdown quantile is ~2× optimistic (realized exceedance 0.082–0.102) — a plug-in bias from seeing only one realized path. Practical reading: any bootstrap drawdown quantile is a *lower bound* on tail severity; multiply the nominal tail probability by ~1.5–5×.
- **CLAIM (Journal of Empirical Finance, 2026, "Don't draw the downs apart"):** among resampling schemes for max-drawdown distributions, the Efron i.i.d. bootstrap systematically underestimates true MDD; the **stationary bootstrap (Politis–Romano 1994, geometric random block lengths)** was the most accurate and robust of the block family — but per the coverage study above, even it does not capture regime-persistence in variance. Ordering of fixes: i.i.d. < block/stationary < regime-conditional.
- **CLAIM (DeltaTrend, already in vault):** regime-switching MC recipe = tag each trade with a regime → per-regime outcome distributions → trade-to-trade transition counts → row-normalize → sample regime-conditionally. He adds one practically important point: **the transition matrix may be estimated from a longer dataset than the trade ledger itself** (e.g. daily vol-regime tags over years), decoupling the matrix's sample size from the strategy's trade count.
- **Steady-state check:** solving \(\pi P = \pi\), \(\sum \pi_i = 1\) gives long-run regime occupancy — a free consistency diagnostic (simulated occupancy should match \(\pi\), and \(\pi\) should roughly match the historical fraction of days in each regime). The Vault already has this machinery in `vault-app/lib/markov-occupancy.ts` (non-overlapping-window transition counts + empirical occupancy).

## Details / mechanics

**Minimal 2-state implementation for a trade-level path MC (concrete enough to code):**

1. **Regime tagging (day level, not trade level).** Tag every *calendar day* in the sample, e.g. `HIGH` if 10-day realized vol of MNQ daily ranges (or VIX close) is above its rolling median, else `LOW`. Day-level tagging lets the matrix be estimated from the full price history (hundreds of days) rather than the 15–40-trade ledger — DeltaTrend's decoupling trick, and the single biggest defense against small-sample noise.
2. **Transition matrix.** Count day-to-day transitions \(n_{ij}\); estimate \(\hat p_{ij} = (n_{ij} + \alpha) / (n_{i\cdot} + k\alpha)\) with Laplace smoothing \(\alpha = 1\) so no transition is estimated at exactly 0 from a short window. For a 2-state chain, report the implied mean regime duration \(1/(1-\hat p_{ii})\) — sanity-check it against eyeballed vol-cluster lengths (typically 5–20 trading days for equity-index vol).
3. **Per-regime trade distributions.** Split the trade ledger's R-multiples by the regime of their entry day. With small ledgers, apply shrinkage: sample from the regime bucket with probability \(w = n_r/(n_r + m)\) (e.g. \(m = 10\)) and from the pooled ledger with probability \(1-w\). This degrades gracefully to i.i.d. reshuffle when a regime bucket is empty.
4. **Path generation.** Draw starting regime from \(\pi\) (or today's known regime); for each simulated trading day: advance the regime by one Markov step, decide whether a trade occurs that day using the *regime-conditional trade frequency* (setups/day observed in that regime — regimes affect frequency, not just quality), then draw the trade outcome from the regime's distribution. Feed the resulting P&L stream through the existing eval/funded barrier logic unchanged.
5. **Outputs to compare vs i.i.d.:** pass rate, bust rate, timeout rate, `E[$/wk]`, worst-drawdown p95, and losing-streak p95. Expect the i.i.d. and regime-aware runs to agree on means and diverge on tails — if they don't diverge, regimes don't matter for this strategy and i.i.d. stays the default (cheapest model wins).

**Why not full Hamilton MLE:** the Hamilton filter estimates regimes and parameters jointly from returns, which is the right tool when regimes are unobservable. For the Vault's purpose an *observable* proxy regime (realized vol / VIX threshold) is preferable: transparent, no local-optima estimation risk, auditable in the ledger, and per the coverage study the payoff comes from *any* persistence in variance being represented, not from optimal regime inference.

**Failure modes to respect:**

- With a 2-state chain and shrinkage, total new free parameters ≈ 2 (matrix) + 1 (threshold) — acceptable. A 3+ state chain on a 30-trade ledger is overfitting surface; don't.
- The regime threshold is itself a researcher degree of freedom — pre-register it (rolling median) rather than sweeping it, or the regime MC becomes another selection-bias engine (see `multiple-testing-selection-bias-sleeves.md`).
- Regime-conditioning fixes the *variance-persistence* channel only. If trade outcomes are autocorrelated *within* a regime (they shouldn't be for a one-trade/day system with independent daily setups), a day/week block bootstrap layered inside regimes handles it — the engine's existing `bootstrap: "day" | "week"` modes are exactly this.

## APPLICATION TO THE VAULT

1. **Priority justified by evidence, not fashion.** The coverage study is the first citable quantification of *when* the Vault's current i.i.d./day/week reshuffle misleads: mean-like outputs (trade EV) are fine; **tail path outputs (worst-drawdown p95, bust rate against the trailing DD, streak p95) are optimistic by roughly 1.5–5× in tail probability if vol regimes persist**. Since eval bust is a barrier event driven by exactly those tails, the regime upgrade is a bust-rate-accuracy fix, not a nice-to-have.
2. **Concrete engine path (post-May backlog, no code edited now):** extend `McParams` with optional `dayRegimes?: Record<string, string>` (date → tag) and a `bootstrap: "regime"` mode; reuse `buildMarkovOccupancy()` for the transition matrix (it already computes `transitionP`); add the shrinkage sampler from §3 above. The golden-test file (`mc-golden.test.ts`) should pin i.i.d. vs regime-aware outputs on a synthetic ledger with planted regime clustering, verifying the regime mode reproduces deeper drawdown tails.
3. **Interim honesty rule (can adopt immediately, zero code):** whenever the scorecard quotes a bootstrap drawdown or bust-rate quantile, annotate it as a *lower bound* on tail severity ("nominal 5% ≈ true 8–25%"). This is a wording change to how MC results are read, consistent with the freeze.
4. **Decision rule for adoption:** tag the May+June Dual46 ledger days HIGH/LOW vol, run both MC modes, and adopt regime-aware as scorecard default only if bust rate or `E[$/wk]` p05 moves by more than the MC's own seed-to-seed noise. If Dual46's outcomes prove regime-flat (possible — the 100-pt cap and fixed 1:5 bound per-trade variance by construction), record that as a finding and keep i.i.d.
5. **Ties to topic 45:** the vol-regime-dependence note (`vol-regime-dependence-setup-frequency.md`) supplies the empirical question of whether Dual46 setup *frequency/quality* actually differs by regime; this note supplies the machinery that consumes the answer.
